# Professional Consulting Standard

用于把金融 PPT 从“排版好看”提升到“专业咨询交付物”。生成、优化、审校复杂金融 deck 时必须读取本文件。

## Core Standard

专业金融咨询 PPT 必须同时通过四类检验：

1. **Decision usefulness**：能帮助受众做决策，而不是只汇报信息。
2. **Analytical rigor**：每个结论有事实、数据、方法或案例支撑。
3. **Traceability**：每个数字、判断、图表和方法假设都能追溯。
4. **Executive readability**：高层读者不听讲解也能理解主线。

## Partner Review Gate

在交付前模拟合伙人审稿。每页回答以下问题：

- 这页回答了哪个管理层/董事会/投委会问题？
- 标题是否是一句可以被证据支撑的判断？
- 主证明对象是否足以支撑标题？
- 这一页的管理含义是什么？
- 如果删掉这页，主线是否断裂？如果不断裂，考虑删页或并页。
- 是否存在“事实很多、洞察不足”的页面？
- 是否存在“结论很强、证据不足”的页面？
- 是否存在敏感判断没有前提、口径或边界？

## Page Decision Test

每页必须落入以下四类之一：

| Page type | Purpose | Must contain |
|---|---|---|
| 决策页 | 推动批准、选择、授权或行动 | 建议、证据、风险、下一步 |
| 诊断页 | 解释问题或机会的原因 | 结论、数据/案例、机制解释 |
| 证明页 | 支撑前文关键判断 | 图表/矩阵/证据链、来源 |
| 方法页 | 说明口径、样本、计算方法 | 定义、范围、限制、影响 |

如果页面不属于这四类，通常不是专业咨询正文页，应删除、移入附录或改写。

## Executive Implication Rule

每个重要页面必须至少包含一个管理含义，常见句式：

- `这意味着……`
- `对管理层的含义是……`
- `因此，本页支持的决策是……`
- `需要进一步确认的前提是……`
- `若该判断成立，下一步应……`

不要只写“数据展示”“情况说明”“分析结果”。必须说清楚它为什么重要。

## Evidence Hierarchy

证据优先级从高到低：

1. 用户提供的原始文件、模型、审计底稿、年报、公告。
2. 权威监管、交易所、央行、统计局、行业协会或公司披露。
3. 可复算的模型和指标计算。
4. 可引用的第三方研究或新闻。
5. 专家判断或项目组假设。

低优先级证据不能单独支撑高强度结论；必须写成假设、待确认或情景。

## Assertion Strength Ladder

根据证据强度选择措辞：

| Evidence | Allowed language |
|---|---|
| 强数据/文件证据 | `显示`、`表明`、`已形成`、`排名为` |
| 多证据交叉支持 | `较可能`、`指向`、`支持判断` |
| 单一证据或推断 | `可能`、`需进一步验证`、`初步显示` |
| 假设或情景 | `在……假设下`、`若……成立`、`情景测算` |

禁止在证据不足时使用“必然”“显著”“决定性”“核心原因”等强措辞。

## Deck-Level Quality Gates

交付前必须检查：

- 执行摘要中的每个结论都能回链正文页。
- 正文页中的每个关键结论都能回链数据、事实或案例。
- 方法论页说明样本、期间、口径、限制和不可比因素。
- 行动建议能回链问题、机会或风险。
- 敏感结论保留前提和适用边界。
- 章节顺序形成“问题/机会 → 证据 → 含义 → 行动”的阅读路径。
- 视觉没有掩盖事实边界；红色、绿色等语义色不制造过度判断。

## Professional Anti-Patterns

避免：

- 只描述现象，不解释原因或含义。
- 只给建议，不证明为什么。
- 用口号式标题替代结论。
- 在摘要页引入正文没有证明过的新结论。
- 用复杂图表掩盖数据口径不清。
- 为了视觉整齐删掉关键前提。
- 把推演、估算、项目组判断写成事实。
- 将“相关性”写成“因果关系”。

      }
    ]
  }
