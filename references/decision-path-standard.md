# Decision Path Standard

用于把金融咨询 PPT 从“提出建议”提升为“说明为什么选择这条路径”。适用于董事会、投委会、管理层、交易谈判、风险处置和战略行动类材料。

## Core Principle

专业咨询交付不能只给一个建议，还要说明：

```text
客户要做什么决定
  -> 有哪些可选路径
    -> 按什么标准比较
      -> 为什么推荐当前路径
        -> 如果不做或做错，会有什么后果
          -> 下一步如何落地
```

如果没有路径取舍，建议容易显得像口号。

## Decision Question Rule

`client_question` 必须尽量写成决策问题，而不是主题问题。

| Weak | Strong |
|---|---|
| `保险公司财务结果如何？` | `新准则下利润改善是否可持续，下一阶段应重点监控哪些指标？` |
| `不良处置行业实践有哪些？` | `在当前监管约束下，哪些不良处置路径更适合本机构优先推进？` |
| `估值是否合理？` | `在基准和压力情景下，当前报价是否仍具备安全边界？` |
| `银行经营表现如何？` | `当前账面盈利是否能转化为估值修复，管理层应优先补哪类短板？` |

## Options Comparison Rule

只要材料中包含建议、方案、路径、行动计划，就必须至少比较 2 个路径：

- `status_quo`：维持现状或原方案。
- `recommended`：推荐路径。
- `alternative`：可选但不推荐路径。

如果只有一个方案，也要说明为什么其他路径不适合。

比较标准必须与客户决策相关，常用维度：

- 收益贡献。
- 风险暴露。
- 资本占用。
- 执行难度。
- 时间要求。
- 数据可得性。
- 监管/合规边界。
- 组织协同成本。

## Recommendation Anatomy

推荐页必须包含：

| Element | Requirement |
|---|---|
| 推荐路径 | 一句话说明选择 |
| 推荐理由 | 2-3 个证据支持 |
| 不选其他路径的原因 | 至少 1 条 |
| 前提条件 | 数据、监管、市场或组织前提 |
| 风险和缓释 | 主要风险 + 缓释动作 |
| 下一步动作 | 责任主体、时间、指标或输出物 |

不要只写“建议采用方案二”。必须写清楚为什么。

## Consequence Of Inaction

客户交付版必须说明“不做会怎样”，但措辞保持稳健：

- `若维持当前口径，管理层可能无法及时区分利润增长来源`
- `若不提前设定阈值，风险暴露可能在季度复盘后才被识别`
- `若未统一样本口径，同业比较结论可能被客户质疑`

禁止使用恐吓式表达，如“必然失败”“严重后果”“巨大损失”，除非有强证据。

## Visual Implication

与决策路径相关的页面优先使用：

- `issue_tree`：把客户问题拆成 3-5 个子问题。
- `option_comparison`：比较不同路径。
- `recommendation_path`：展示推荐路径及阶段。
- `risk_mitigation_matrix`：说明风险和缓释动作。
- `management_action_plan`：明确责任、时间、指标、输出。
- `decision_one-page`：把推荐、证据、风险和下一步压缩为一页。

详见 `references/consulting-page-archetypes.md`。

## Decision Path QA

构建前检查：

- 是否有明确 `decision_to_make`。
- 是否至少比较两个路径，或说明为什么其他路径不可取。
- 推荐路径是否回链到证据页。
- 推荐理由是否包含收益、风险、执行和边界。
- 是否说明“不做会怎样”。
- 下一步动作是否有责任、时间、指标或输出。
